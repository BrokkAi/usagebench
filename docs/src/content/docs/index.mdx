<h2 class="landing-headline">Usage lookup, compared at the source-location level</h2>

UsageBench measures whether code-navigation analyzers return the same reviewed
declarations, usages, and navigation targets. The current snapshot compares
Bifrost with ten language servers across small, checked-in fixtures.

These are contract results, not a general analyzer leaderboard. A policy-only
near miss contains every required location but also includes an import,
re-export, or export-metadata location that UsageBench excludes. A hard
disagreement can combine missing usages, broader symbol families, or a different
navigation target.

<div class="execution-contrast">
  <span>EXECUTION MODEL</span>
  <strong>Static on one side; full build context on the other.</strong>
  <p>
    Bifrost never executes the code under analysis or invokes its project build.
    Language servers may receive configured toolchains, restore dependencies,
    import projects, and build fixtures when their normal semantic model requires
    it. The comparison gives each analyzer the environment it needs; Bifrost's
    results remain build-independent.
  </p>
</div>

The denominators differ because Bifrost can address one zero-width Python module
selector that the generic LSP runner cannot express as a cursor request. Seven
not-planned cases are outside both denominators; three LSP-unsupported cases and
two Bifrost-unsupported cases are also outside them.

All 136 exact Bifrost outcomes are expected passes in the current baseline. The
five expected gaps are the cases that remain non-exact in the current run.

<div class="evidence-note">
